# SPI-100: Scoped Reserved Principal Keys

SPI-100 defines a stateless way to encode protocol-local identifiers as
Internet Computer `Principal` values without colliding with real users,
canisters, anonymous callers, or external ledgers.

The primitive is a scoped reserved principal:

```text
Key(scope, local_id) = bytes(scope) || u32_be(local_id) || 0x7f
```

`scope` names the authority or namespace. `local_id` names one entry inside that
namespace. The final byte `0x7f` places the output in the IC reserved-principal
class.

SPI-100 intentionally has no magic prefix, version byte, kind byte, hash, or
registry slot. The full scope principal is embedded directly, and the encoder
returns `null` if it does not fit.

## Goals

SPI-100 is meant to give SR9 protocols one reusable key shape for local
protocol objects:

- **Virtual assets:** a canister can represent pool shares, vault shares, DAO
  positions, or other internal balance classes as `Principal` keys. Example:
  `Key(dexCanister, poolId)` is the virtual principal for one DEX pool.
- **Delegated local accounts:** a principal can control many local accounts
  inside another canister. Example: `Key(owner, accountId)` is account
  `accountId` controlled by `owner`.
- **Client interoperability:** balances can be returned as `Principal -> Nat`,
  so clients can use one account-key type for real ledgers and local virtual
  balances.
- **No allocation registry:** keys are deterministic and stateless. Protocols do
  not need a global table mapping owners to compact slots, so there is no
  registry exhaustion or cleanup surface.
- **Human and tool compatibility:** reserved principals still have the normal
  IC principal text representation, so they can be logged, displayed, compared,
  sorted, and used as map keys.

## IC Principal Facts

The Internet Computer interface specification defines a principal as an opaque
binary blob with length between `0` and `29` bytes. Principals can identify
canisters, users, and future IC concepts. The specification also says there is
intentionally no general mechanism to distinguish canister ids from user ids by
looking at the bytes.

The IC spec defines several special principal forms:

- opaque ids are generated by the IC and typically end in `0x01`;
- self-authenticating ids have the form `H(public_key) || 0x02`;
- derived ids have the form
  `H(|registering_principal| || registering_principal || derivation_nonce) || 0x03`;
- the anonymous id is the one-byte principal `0x04`;
- reserved ids have the form `blob || 0x7f`, where `0 <= |blob| < 29`.

Reserved ids are the SPI-100 foundation. The IC spec says they explicitly
indicate that the blob does not address any canister or user, and that when the
IC creates a fresh id, it never creates a reserved id.

Sources:

- IC Interface Specification, Principals:
  <https://internetcomputer.org/docs/references/ic-interface-spec#principals>
- IC Interface Specification, Special forms of Principals:
  <https://internetcomputer.org/docs/references/ic-interface-spec#special-forms-of-principals>

## Format

The byte layout is:

```text
0..n-1          n..n+3             n+4
scope bytes     local id u32-be    reserved marker 0x7f
```

Definitions:

```text
B(p)      = raw principal bytes of p
U32(i)    = four-byte big-endian representation of i
Key(s, i) = B(s) || U32(i) || 0x7f
```

Valid inputs:

```text
1 <= |B(scope)| <= 24
0 <= local_id <= 2^32 - 1
scope is not anonymous
scope is not reserved
```

The maximum scope size is `24` bytes because IC principals are at most `29`
bytes:

```text
|Key(scope, local_id)| = |B(scope)| + 4 + 1
|Key(scope, local_id)| <= 29
therefore |B(scope)| <= 24
```

## Semantics

### Virtual Assets

For virtual assets, `scope` is the canister or protocol principal that owns the
namespace.

```text
poolShare = Key(dexCanisterPrincipal, poolId)
```

A client can decode `poolShare` and learn:

```text
scope = dexCanisterPrincipal
id    = poolId
```

That makes the key self-describing enough for routing and display. A wallet or
indexer can recognize that the virtual asset belongs to the DEX canister and
ask that canister what `poolId` means.

SPI-100 does not encode an asset kind. The `local_id` namespace belongs to the
scope. If one canister uses SPI-100 for multiple families, such as pools and
vault shares, it must allocate disjoint id ranges or define its own mapping from
ids to meanings.

Authorization for virtual assets is protocol-defined. For example, a DEX changes
pool-share balances only through liquidity rules. No external caller should be
able to authenticate as the virtual asset key itself.

### Delegated Accounts

For delegated accounts, `scope` is the owner/controller principal.

```text
account = Key(ownerPrincipal, accountId)
```

The authorization rule is:

```text
Authorized(caller, account) =
  usable(caller) and Decode(account).scope = caller
```

In implementation terms, a canister must decode the account key and compare the
embedded scope with the IC-authenticated `caller`. Possessing the key value is
not authorization. Any caller can pass any `Principal` as an argument.

Example:

```text
account = Key(canisterA, 7)

caller = canisterA  -> authorized
caller = canisterB  -> rejected
caller = userX      -> authorized only if account embeds exactly userX
caller = anonymous  -> rejected
caller = reserved   -> rejected
```

The protocol must never check `caller == account`. SPI-100 account keys are
reserved principals; control comes from authenticating as the embedded `scope`.

### Unified Control Check

Most protocol code should not branch manually between normal principals and
SPI-100 principals. The common rule is:

```text
Control(principal) =
  if principal is SPI-100:
    { controller = Decode(principal).scope; id = ?Decode(principal).id }
  else if principal is direct:
    { controller = principal; id = null }
  else:
    null
```

`direct` means a non-empty principal that is neither anonymous nor reserved. It
can be a user, a canister, or any future normal IC principal form.

The gate for a public method is:

```text
Authorize(caller, principal) =
  if caller is direct and Control(principal).controller = caller:
    Control(principal)
  else:
    null
```

This gives one ergonomic authorization path:

- direct account: `Authorize(caller, caller)` succeeds with `id = null`;
- delegated account: `Authorize(owner, Key(owner, id))` succeeds with
  `id = ?id`;
- arbitrary caller passing someone else's key fails;
- anonymous and reserved callers fail;
- reserved values that are not valid SPI-100 keys fail.

The returned `ControlInfo` is both the gate and the decoded data. Protocol code
can use it to decide whether the caller controls the principal and, when the
principal is SPI-100, which `local_id` was selected.

## Safety Properties

These properties are what SPI-100 relies on. Some are IC protocol facts, and
some are consequences of the byte format.

### 1. Valid Principal Size

For every accepted input:

```text
|Key(s, i)| = |B(s)| + 5
|B(s)| <= 24
therefore |Key(s, i)| <= 29
```

So every produced key is a valid-size IC principal blob.

### 2. Reserved-Class Membership

For every produced key:

```text
lastByte(Key(s, i)) = 0x7f
```

Therefore the key is in the IC reserved-principal class `blob || 0x7f`.

### 3. No Collision With IC-Issued Identities

The IC spec states that fresh IC-created ids are never reserved ids and that
reserved ids do not address any canister or user.

SPI-100 produces only reserved ids:

```text
SPI100 = { B(s) || U32(i) || 0x7f }
IssuedICIds ∩ ReservedIds = empty
therefore SPI100 ∩ IssuedICIds = empty
```

So a SPI-100 key cannot be the principal of a real canister, a ledger canister,
a self-authenticating user, a derived id, or the anonymous principal.

### 4. No Direct Possession

Because SPI-100 keys are reserved ids, and the IC spec says reserved ids do not
address users or canisters, no normal IC identity can authenticate a message
with:

```text
caller = Key(scope, local_id)
```

This is why virtual balances can safely be keyed by SPI-100 principals while
still rejecting them as external ledger principals or authenticated callers.

### 5. Injective Scoped Encoding

The encoder embeds the full scope bytes and a fixed-width local id. It does not
hash or truncate. If two SPI-100 keys are equal:

```text
Key(s1, i1) = Key(s2, i2)
B(s1) || U32(i1) || 0x7f = B(s2) || U32(i2) || 0x7f
```

The final byte is the same reserved marker, the previous four bytes are the
fixed-width id, and the remaining prefix is the complete scope:

```text
U32(i1) = U32(i2)  -> i1 = i2
B(s1) = B(s2)      -> s1 = s2
```

Therefore:

```text
Key(s1, i1) = Key(s2, i2)  ->  s1 = s2 and i1 = i2
```

Different scopes or different ids produce different keys.

This is the property that prevents two different principals from being assigned
the same delegated account key without a stateful registry.

### 6. Decode Roundtrip

For every accepted input, a conforming decoder must satisfy:

```text
Decode(Key(scope, local_id)) = { scope; id = local_id }
```

This is the practical property clients and protocols need. The current SR9 code
proves this through the SPI-100 contracts and `VirtualPrincipalObservers.sr9`.
The SPI module itself is untrusted; the low-level byte/principal facts live in
the SR9 core library.

### 7. Delegated Caller Binding

For delegated accounts:

```text
Authorized(caller, Key(scope, id)) -> caller = scope
```

The IC supplies `caller` from message authentication. A different caller cannot
satisfy the equality check unless it is authenticated as the embedded scope.

This is the security boundary for delegated accounts. Passing or discovering a
delegated account key does not grant control over it.

### 8. Unified Direct-Or-Scoped Authorization

The high-level `Authorize` helper preserves the two intended ownership cases:

```text
Authorize(c, p) = ?info  ->  info.controller = c
```

For direct principals:

```text
Control(p) = { controller = p; id = null }
Authorize(c, p) succeeds  ->  c = p
```

For SPI-100 principals:

```text
Control(Key(s, i)) = { controller = s; id = ?i }
Authorize(c, Key(s, i)) succeeds  ->  c = s
```

So application code can call one helper for both ordinary accounts and
delegated local accounts, while still getting the decoded id when there is one.

## Benefits

SPI-100 gives protocols a compact primitive with useful operational properties:

- one `Principal` key type for real ledgers, virtual assets, and delegated local
  accounts;
- deterministic key generation with no allocation race;
- no stateful registry and therefore no registry DoS or migration burden;
- direct compatibility with ordered maps using `Principal.compare`;
- easy client decoding for balances and positions;
- stable keys across canister upgrades because the encoded scope bytes do not
  change;
- clear rejection rules for external ledgers, callers, anonymous principals, and
  reserved principals.

For protocols like a DEX, DAO, vault, or lending pool, this lets public balance
APIs expose local positions in the same shape as token balances:

```text
Principal -> Nat
```

The client does not need a separate `Text` key system or a protocol-specific
variant just to display local positions.

## Limits And Risks

SPI-100 is intentionally narrow. Implementations must account for these limits.

### Future Principal Size

SPI-100 with a fixed `u32` local id supports scopes whose raw principal bytes
are at most `24` bytes.

If the IC protocol later starts issuing new canister principals larger than 24
bytes, those new canisters will not be able to use this exact SPI-100 format
with a 32-bit local id. `encode` must return `null` rather than hashing,
truncating, or silently using a weaker key.

This does not break existing SPI-100 deployments. A canister principal is stable
for the lifetime of that canister. Existing keys embed the old canister id bytes,
and those bytes continue to decode the same way after upgrades or future IC
changes.

### Full-Length Principals

The IC allows principals up to 29 bytes. A full 29-byte scope cannot be embedded
inside another 29-byte reserved principal:

```text
29 byte scope + 4 byte id + 1 byte 0x7f > 29
```

There is no stateless, collision-free way to encode every possible IC principal
plus even a 1-byte local id inside one IC principal while preserving the full
scope. Supporting full-length owners would require a separate protocol, usually
with a registry. SPI-100 avoids that registry by rejecting non-fitting scopes.

### Local Id Namespace

SPI-100 has no kind byte. This preserves scope headroom, but it means all
meaning is inside the scope's `local_id` namespace. Protocols that need several
families of objects must reserve id ranges or define an internal id allocation
scheme.

### Not External Ledgers Or Call Targets

SPI-100 keys are local identifiers. They must not be registered as ICRC ledgers,
called as canisters, or accepted as authenticated callers. External ledger
registration should reject reserved principals.

### Trusted Runtime Boundary

The safety argument assumes faithful principal/blob conversion:

```text
Principal.fromBlob(B(p)) = p
Principal.toBlob(Principal.fromBlob(b)) = b
```

The current SR9 implementation uses Motoko/IC principal blob conversion for this
behavior. Some of these roundtrip facts are still trusted boundaries in SR9
until core exposes stronger principal/blob axioms.

The SPI-100 module itself contains no `trusted` functions and does not import
`mo:⛔`. Its trust boundary is pushed into core primitives that already represent
runtime facts:

- `Principal.toBlob` and `Principal.fromBlob` expose the raw IC principal bytes;
- `Blob.get`, `Blob.slice`, `Blob.concat`, `Blob.getNat32BE`, and
  `Blob.appendNat32BEByte` expose bounded byte operations with verifier-visible
  size and layout facts;
- `Principal.encodeScopedReserved` and `Principal.decodeScopedReserved` expose
  the scoped-reserved-principal roundtrip contract used by SPI-100.

The ergonomic control helpers are intentionally small wrappers around this
decoder. The BMap examples keep their map invariants pure and inline so
protocol code can prove ordering and balance-preservation properties without
re-proving principal byte facts at every call site.

### IC Reserved-Id Semantics

SPI-100 depends on the IC specification continuing to treat reserved principals
as non-addressing ids that are never freshly issued by the IC. If the IC ever
changed reserved ids into addressable or issuable principals, protocols using
SPI-100 would need a new safety review.

## API Footgun Avoidance

The module exposes a narrow public API so protocol code has one obvious safe
path:

- Use `encode(scope, id)` to create a SPI-100 principal.
- Use `decode(key)` only for structural decoding and display. It does not
  authorize mutation.
- Use `resolveControl(key)` only for routing and introspection. It tells which
  principal controls a key, but it does not prove that the current caller is
  that principal.
- Use `authorize(caller, key)` as the mutation gate when a user may act through
  either their normal principal or a delegated SPI-100 principal.
- Use `canControl(caller, key)` only when the boolean answer is enough.
- For scoped-only flows, call `authorize(caller, key)` and then require
  `ControlInfo.id != null`. This prevents a direct principal from accidentally
  passing a path that expects a delegated account key.

The module intentionally does not expose separate public helpers like
`isScopedBy(scope, key)` or scoped-only authorization aliases. Those helpers are
easy to compose incorrectly. `authorize` is the canonical gate because its
postcondition says that a successful result has `controller == caller`.

## Implementation Rules

Implementations using SPI-100 must follow these rules:

- Encode only with `Key(scope, local_id) = bytes(scope) || u32_be(local_id) || 0x7f`.
- Reject anonymous and reserved scopes.
- Return `null` when the scope does not fit.
- Never hash or truncate a scope principal.
- Treat the same `(scope, local_id)` pair as the same key.
- Treat different `(scope, local_id)` pairs as different keys.
- Reject SPI-100 keys as external ledger principals.
- Do not expect or accept `caller == spi100Key`.
- For delegated or direct user accounts, prefer `authorize(caller, key)` or
  `canControl(caller, key)` over ad hoc checks.
- Treat `resolveControl(key)` as a decode/routing helper only. It tells who
  would control a key, but it does not prove that the current caller is that
  controller.
- Treat `decode(key)` as structural decoding only. It does not authorize
  mutation.
- For virtual assets, define authorization in the owning protocol's rules.

## SR9 Draft

`VirtualPrincipal.sr9` contains the draft module:

- `encode(scope, id)` encodes the SPI-100 key.
- `decode(key)` returns `{ scope; id }` when the key has the SPI-100 shape.
- `resolveControl(key)` returns `{ controller; id }` for either a direct
  principal or a valid SPI-100 key.
- `authorize(caller, key)` combines gating and decoding. It returns
  `{ controller; id }` only when `caller` controls `key`.
- `canControl(caller, key)` is the boolean form of `authorize`.
- `isDirectPrincipal(principal)` checks the normal-principal branch used by
  `resolveControl`.
- `isVirtual(key)` checks whether decoding succeeds.
- `isReserved(principal)` checks the IC reserved class byte.
- `isAnonymousPrincipal(principal)` checks the anonymous principal shape.
- `isUsableEmbeddedPrincipal(principal)` checks the scope constraints.

`examples/VirtualPrincipalMapDemo.sr9` demonstrates the storage pattern:

- virtual pool-style keys scoped by `Runtime.selfId()`;
- delegated keys scoped by the authenticated caller;
- `BMap.BMap<Principal, Nat>` keyed directly by the produced principals;
- credit, debit, move, read, list, and delegated authorization paths.

The `examples/` directory adds verification-oriented usage patterns:

- `AccountBook.sr9` is a reusable BMap account book keyed by `Principal`. It
  uses `authorize` for direct and delegated-account credit/debit/transfer.
- `UnifiedAccountLedgerDemo.sr9` is an actor where callers can use their normal
  principal or their scoped SPI-100 account principals through the same public
  methods.
- `ProtocolAssetBook.sr9` is a reusable module for protocol-owned virtual asset
  keys, such as pool-share ids scoped by the canister principal.
- `ProtocolVirtualAssetDemo.sr9` is an actor where a controller mutates
  protocol-owned virtual balances by protocol rule rather than caller ownership.

The `proofs/` directory contains observer proofs:

- `VirtualPrincipalObservers.sr9` proves accepted `encode(scope, id)` values
  decode back to the same `{ scope; id }`;
- if two accepted encodings are equal, the observer proves both scopes and ids
  are equal;
- as direct corollaries, the observer proves same-scope/different-id and
  same-id/different-scope encodings do not collide.

The module currently verifies with SR9 without any module-local `trusted`
helpers. Low-level byte and principal conversion assumptions are isolated in the
core library boundary described above.
